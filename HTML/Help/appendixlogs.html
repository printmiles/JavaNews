Within the application you need to set a logging directory along with a logging level. While the application runs it generates events. These can range in severity and normally don't need to be seen. However, if there are problems while running the application then the log files are useful to finding exactly what the problems is and where it lies. You should make the logging directory used by the application somewhere that is easy to locate and is always available (avoid using network drives if they might not be available).

Organisation of the Files

If you were to look into the log directory, the first reaction might be "there's an awful lot of files in there", or, "it looks complicated, I'll leave it alone".

Each of the files contained within the directory represent a particular class within the application. A class can be thought of as a chunk of code responsible for a set of tasks within the application as a whole. The log files for JavaNews can be distinguished from other XML files from the fact that they end .log.xml.

All of the .log.xml are named after a fully qualified package and class name followed by a sequential number for conflicts. For JavaNews all of the packages are in the javanews package. This forms a common start for all of the file names. After this is the next sub-package name and the class name, separated by a full-stop (for example javanews.gui.Client is the Client class within the gui package within the application). The sub-packages are organised as follows:

Name Purpose Qualified Name
database Used to hold classes used to connect to or work with the back-end database through JDBC and/or SQL. .database
events Used to hold event classes used to perform actions within the application. .events
gui (Subclass of events) Used to hold GUI events triggered from the user working with the application. .events.gui
table (Subclass of events) Used to hold data events relating to tables and the data within them being changed. .events.table
gui Used to hold the major GUI classes. Mainly the main application window and the splash screen shown during start-up. .gui
internalframe (Subclass of gui) Used to hold the classes of windows shown within the main application window. These include configuration, help and data windows. .gui.internalframe
object Used to hold objects commonly used by other classes within the application. .object
print (Subclass of object) Used to hold classes required specifically for printing. .object.print
table (Subclass of object) Used to hold classes used to hold and display data within tables so they can be displayed on-screen. .object.table
run Used to hold classes needed to start the application. .run

For more information about the purposes of each class within the application, you can refer to the Javadoc.

Viewing the Files

The log files are also used to record any messages received from the PSE network and can be found in some of the files within the log directory. If you use a file browser to explore the logging directory you will see many different files. They can be opened in any application that recognises XML content. Microsoft's Excel is good for compiling multiple XML files, while Apache Chainsaw is able to view log files as well.

The document type definition (DTD) for log files can be viewed on the Sun Microsystems website and it gives information about the contents of the XML file and the purpose for each of the fields.

Microsoft Excel

Excel allows for records to be imported and then sorted and filtered using tools familiar to most.

Multiple files must be manually cut and paste into a single file to follow the program execution.

Apache Chainsaw

Apache chainsaw allows for XML log files to be viewed and sorted similar to Excel. However, it does have problems accessing large files.

Other Applications

Other applications can be used to open XML files, notepad is one and all will display the content of these text based files. For more information on the information contained within a log file, refer to the DTD at the Sun site.
